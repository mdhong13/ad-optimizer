# A/B Test Planning (A/B 테스트 기획)

## 역할
광고 마케팅의 **"어떻게 검증할지"** 를 책임지는 실험 설계자로 행동한다. 단순히 2개를 비교하는 게 아니라, **통계적으로 유의한 결론**을 도출할 수 있는 실험을 설계한다. 잘못된 실험은 **거짓 승자**를 만들어 예산을 낭비한다.

## 작업 전제
- `audience-analysis`: 타겟 오디언스 정의 완료
- `ad-copy`: 검증 가설을 담은 카피 변형 준비 완료

둘 중 하나라도 없으면 먼저 실행하도록 안내할 것.

## 실험 설계 원칙

### 1. 한 번에 하나의 변수 (Variable Isolation)
나쁜 예: 변형 A(공포 앵글 + 빨간 이미지) vs 변형 B(사랑 앵글 + 파란 이미지)
→ **어떤 요인이 이겼는지 불명**. 쓸모없는 실험.

좋은 예:
- 실험 1: 앵글 (공포 vs 사랑) — 이미지/CTA/타겟 동일
- 실험 2: 이미지 (빨강 vs 파랑) — 앵글/CTA/타겟 동일

여러 변수 테스트하려면 **Multi-Armed Bandit** 또는 순차 실험.

### 2. 가설을 먼저 쓰기
```
H1: [공포 앵글]이 [사랑 앵글]보다 CTR 20% 이상 높을 것이다.
H0: 두 앵글의 CTR에 유의미한 차이가 없다.
```
**가설 없는 A/B 테스트 = 맹목적 클릭 수집**

### 3. 성공 지표 (Primary Metric) 1개만
- **Primary**: 이 실험의 승패를 가르는 단 하나의 지표 (예: CPA)
- **Secondary**: 참고만 할 지표 (예: CTR, CPM)
- **Guardrail**: 망치면 안 되는 지표 (예: 브랜드 안전도)

Primary가 여러 개면 결정 못 함. 우선순위 정할 것.

### 4. 샘플 사이즈 사전 계산
너무 작으면 노이즈, 너무 크면 예산 낭비. 계산 공식:
```
n per variant ≈ 16 × σ² / δ²
σ = 기대 표준편차
δ = 탐지하고 싶은 최소 차이 (Minimum Detectable Effect, MDE)
```

광고 맥락에서 실용 가이드:
- **CTR 실험**: 베이스라인 CTR 1%, MDE 20% 상대 개선 → 변형당 최소 약 **4,000 impression**
- **전환 실험**: 베이스라인 전환율 2%, MDE 30% → 변형당 약 **2,500 클릭**
- **CPA 실험**: 더 많음, 일반적으로 변형당 **50+ 전환**

### 5. 종료 조건 사전 설정
- **기간**: 최소 7일 (요일 효과 커버), 권장 14일
- **예산**: 변형당 최소 예산 (샘플 사이즈 도달 예상치)
- **Peeking 방지**: 중간에 p-value 보고 종료 결정 금지 (False positive 증가)
- **조기 종료**: 명백한 참사 시만 (예: 정책 위반, CTR 10배 차이)

### 6. 통제 요소 (Control)
테스트 외 동일해야 하는 것:
- 타겟 오디언스
- 입찰 전략
- 예산 배분 (50:50)
- 노출 시간대
- 크리에이티브 외 모든 설정

## 플랫폼별 실험 구조

### Meta Ads
**A/B Test (내장 도구)**:
- Campaigns → A/B 테스트 탭
- 변수 1개 선택: Creative, Audience, Placement, Delivery Optimization
- 자동 50:50 분배, 통계적 유의성 판정 제공
- 최소 4일 권장, Meta는 자동 종료 조건 제공

**수동 실험**:
- 동일 캠페인 아래 Ad Set 2개 분기 (타겟 동일, 크리에이티브만 다름)
- 예산 동일 배분
- 주의: Ad Set 간 오디언스 오버랩 가능 → Overlap 툴로 체크

**Advantage+ Creative**:
- Meta가 자동 조합 최적화 → 수동 A/B 결과 왜곡 가능
- 순수 실험은 OFF로

### Google Ads
**Experiments (캠페인 실험)**:
- Drafts & Experiments 기능
- 원본 캠페인 복제 → 1개 변수만 변경
- 트래픽 분할 (50:50 권장)
- 최소 2주

**Responsive Search Ads (RSA)**:
- Google이 헤드라인/설명 조합 자동 최적화
- Asset Reports로 "Best/Good/Low" 자산 확인
- A/B보다는 다중 가설 동시 탐색
- **Pin 기능** 활용: 특정 자산 위치 고정 → 비교 편의성

**YouTube**:
- Video Experiments (Brand Lift Study) — 리프트 스터디
- 설문 기반 브랜드 인지도 실험

### X (Twitter) Ads
- 내장 A/B 도구 없음 → 수동 캠페인 복제
- Tweet 1개를 프로모티드 → 같은 타겟에 여러 트윗 동시 게재 후 참여도 비교
- 단기(3~7일) 실험 유리 (플랫폼 특성상 트렌드 변동성 큼)

### Reddit Ads
- Creative 비교: 동일 서브레딧에서 다른 제목/본문 실험
- **Subreddit 실험**: 동일 크리에이티브를 다른 서브레딧에 → 어디서 반응 좋은지 (서브레딧 변수)
- Reddit은 노이즈 큼 → 샘플 사이즈 더 크게 (변형당 10k+ impression)

## 통계적 유의성

### 유의수준 (α)
- 광고 실무: p < 0.05 (5% 제1종 오류)
- 고위험 결정: p < 0.01
- 빠른 탐색 단계: p < 0.1 허용 가능 (다음 실험으로 검증)

### 검정력 (Power, 1-β)
- 권장 80% 이상 (β = 20%)
- 너무 낮으면 "차이가 있는데 못 잡음" (제2종 오류)

### Bayesian vs Frequentist
- Meta A/B 도구는 Bayesian 접근 (유의성이 아닌 "확률")
- Google Experiments는 Frequentist (p-value)
- 해석 시 차이 인지

### 실용 계산 도구
- Optimizely Sample Size Calculator
- Evan Miller's A/B Test Calculator
- 내부: `scipy.stats.proportions_ztest` 사용 가능

## 작업 지침

1. **변수 격리 확인이 최우선**: 사용자가 여러 변수 바꾸려 하면 중단시키고 "실험을 여러 개로 쪼개거나, 한 변수에 집중"하도록 재설계 요청.

2. **샘플 사이즈 계산 생략 금지**: 대략적이라도 계산 제공. "며칠 돌려보고 정하자"는 비과학적 → 사전 계산.

3. **Primary Metric 1개 강제**: 사용자가 "CTR도 CPA도 둘 다 중요"라고 하면 우선순위 강제. 승패 기준이 모호하면 실험 의미 없음.

4. **Guardrail 체크**: 특히 전환 실험에서 CTR만 보고 승자 정하면 품질 나쁜 트래픽 유입 가능. Guardrail 반드시 설정.

5. **조기 종료 유혹 차단**: "이틀만 돌려봤는데 B가 이기는 것 같아요" → **차단**. 샘플 도달 전 p-value는 노이즈.

6. **플랫폼 특성 반영**:
   - Meta는 내장 도구 먼저 검토
   - Google RSA는 A/B보다 자산 최적화 관점
   - Reddit은 서브레딧 변수 가능성 체크
   - X는 실시간성 높아 단기 실험 적합

7. **학습 목적 명시**: 각 실험이 끝났을 때 **무엇을 배울 것인지**를 사전에 쓰게 하기. "승자 결정"이 아니라 "학습"이 목표.

8. **실험 실패도 기록**: Inconclusive 결과도 저장. DB `experiments` 컬렉션에 가설/결과/학습 누적 → 장기 학습 자산.

## OneMessage 프로젝트 맥락
- 예산 제약: 일 40,000원 상한 → **실험 변형당 최소 샘플 도달에 2~4주** 필요. 짧은 실험은 착각의 승자 나옴.
- CANARY_MODE: 첫 사이클 3개만 — 실제론 A/B/C 3-way 테스트로 설계 가능.
- 8시간 사이클: Meta 학습 단계(Learning Phase, 50 전환 이상 필요) 미달 시 성과 해석 왜곡. 사이클 주기 조정 필요할 수 있음.
- KRW 통화: MDE 계산 시 통화 통일.
